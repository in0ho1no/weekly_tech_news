# 政府AIが無料公開、でもライセンスには注意が必要な話

## きっかけ: 政府AIが無料公開された

2026年4月24日、デジタル庁が政府AI「源内（げんない）」の一部をオープンソースとして公開しました。  
江戸時代の発明家・平賀源内と「Gen AI（生成AI）」を掛け合わせた名前で、全府省庁の職員約18万人への展開を目指している政府専用の生成AI基盤です。  

農林水産省では、通常2ヶ月かかっていた稲作調査の集計作業が3日で完了したという実績もあります。  
公開されたソースコードはMITライセンスのもとで商用利用も認められており、「タダで使えるのか」と反応した方もいたかもしれません。  

今日はこのニュースをきっかけに、OSSとライセンスについて整理してみます。  

:::note
源内はデジタル庁が構築・運用する政府専用の基盤です。
個人で利用するには必要な前提知識が多岐にわたります。
よって、本記事内では源内そのものは紹介しません。
:::

## そもそもOSSとは

### OSSの概要

OSSとは「Open Source Software」の略で、ソースコードが公開されているソフトウェアのことです。  

通常、製品として配布されるソフトウェアはソースコードが非公開ですが、  
OSSはそれを誰でも見られる形で公開しています。  
ソースコードが見えることで、誰でも動作を確認・改良・再配布できる点がOSSの特徴です。  

### 「公開されている」と「無料」は別の話

OSSに関してよくある誤解のひとつが、「公開されているから無料で何でも使える」というものです。  
ここで整理しておきたいのは以下の点です。  

- 「ソースコードが公開されている」こと: OSSかどうかという性質の話
- 「どのような条件で使えるか」: ライセンスの話

この2つは別の話です。  
OSSであっても、使用条件は各ソフトウェアのライセンスによって定められています。

## OSSとライセンスは別の話

### ライセンスとは「使用条件の契約書」

ライセンスとは、そのソフトウェアを「どのような条件のもとで使っていいか」を示した文書です。  
「使用・改変・再配布に関するルールが書かれた契約書」と捉えると分かりやすいでしょう。  

ライセンスの条件はソフトウェアによって異なります。  
「著作権の表示さえすれば自由に使える」ものもあれば、「改変して配布する場合は同じ条件で公開する義務がある」ものもあります。  

OSSであっても、ライセンスの条件を無視して使った場合は著作権侵害になる可能性があります。  

### 源内のライセンスを例に

源内のWebインターフェース部分はMITライセンスで公開されています。  
MITライセンスは、ライセンスの中でも条件が少なく自由度が高い部類に入ります。  
主な条件は「著作権表示を残す」ことだけで、商用利用・改変・再配布が認められています。  

だからといって「MITだから業務でも自由に使える」とはなりません。  
その理由を次のセクションで説明します。  

:::info
ドキュメント部分はCC BY 4.0という別のライセンスが適用されており、
利用の際はデジタル庁についてのクレジット表記が必要となります。
同じソフトウェアでも、部分によってライセンスが異なることがある点も覚えておくとよいでしょう。
:::

## 業務・社内における基本姿勢

### 社内ルールが絶対の大前提

ここは明確にしておきたい点です。

**業務や社内の開発において、ライセンスを確認したからといって、そのソフトウェアを使っていいことにはなりません。**

社内で使用できるソフトウェアは、会社が定めたルールに従うことが大前提です。  
OSSであっても、商用利用可能なライセンスであっても、それは「法的に問題がない」という話に過ぎません。  
会社として許可しているかどうかは、また別の判断です。  

:::caution
業務で使用するソフトウェアについては、社内ルールに従ってください。
「OSSだから」「MITライセンスだから」という理由で独断で利用することはやめましょう。
:::

### なぜ勝手に使ってはいけないのか

組織に属して仕事をするということは、そのルールのもとで行動することを意味します。  
技術的な観点から補足すると、許可されていないソフトウェアが社内に増えると以下のような問題が生じます。

- 情報システム部門が社内のソフトウェアを把握・管理しきれなくなる
- セキュリティ上の脆弱性があっても、存在すら気付かれない状態になる
- ライセンス違反が発生しても、会社として対応できない

個人が「便利だから」と持ち込んだ一本のソフトウェアが、組織全体のリスクになりうるのです。  
情報システム部門が管理する意味は、こうしたリスクを組織として制御するためにあります。  

## 個人で開発する場合の話

### 個人利用時は自分で管理する

業務の話とは切り離して、個人で開発・学習を行う場合の話をします。  

個人でOSSを使う場合は、自分でライセンスを確認する必要があります。  
「なんとなく使っていた」では済まないケースもあります。  
代表的なライセンスの特徴は紹介しますが、自分で確認することは怠らないようにしましょう。  

### 代表的なライセンスの種類

| ライセンス | 主な特徴 | 注意点 |
| --- | --- | --- |
| MIT | 著作権表示さえすれば商用利用・改変・再配布が自由 | 著作権表示の省略は不可 |
| Apache 2.0 | MITに近い自由度。特許に関する条項がある | 特許絡みの利用は要確認 |
| GPL | 改変・配布する場合、同じGPLで公開する義務がある | 成果物の公開義務に注意 |

:::tip
GitHubなどでOSSのソースコードを見ると、
多くのリポジトリはそのルートディレクトリに「LICENSE」というファイルがあります。
まずそのファイルを確認する習慣をつけておきましょう。
:::

:::caution
OSSの多くには『無保証』というルールがあります。
そのソフトが原因でデータが消えたりシステムが止まったりしても、作者は責任を取ってくれません。
企業がOSSの利用を慎重に判断するのは、こうした『責任の所在』を明確にするためでもあります。
:::

### ライセンス確認の習慣を持つ

ライセンスは難解な法律文書のように見えますが、代表的なものは内容がよく整理されています。  
「このソフトはどのライセンスか」を調べる習慣だけでも、大きな違いになります。

また、複数のOSSを組み合わせて使う場合は、それぞれのライセンスを個別に確認する必要があります。  
「どれか一つ確認したからOK」とはならない点も覚えておきましょう。  

:::warning
GPLライセンスのOSSを組み込んだソフトウェアを配布する場合、
自分のコードもGPLで公開する義務が生じる可能性があります。
配布を伴う場合は特に慎重な確認が必要です。
:::

## まとめ

今日のポイントを整理します。

- OSSについて
  - OSSとは、ソースコードが公開されているソフトウェアのこと
  - 「公開されている」ことと「無料で何でも使える」は別の話

- ライセンスについて
  - ライセンスは「使用条件の契約書」
  - OSSであっても、ライセンスの条件を守る必要がある
  - 代表的なものにMIT・Apache 2.0・GPLがある

- 業務での基本姿勢
  - 社内ルールが絶対の大前提
  - ライセンスを確認しても、会社が許可していなければ使えない

- 個人開発での基本姿勢
  - 使用するOSSのライセンスを自分で確認する
  - 複数のOSSを組み合わせる場合は、それぞれ個別に確認する

業務での話と個人での話を混同しないことが重要です。  
この2つを区別して考える習慣が、トラブルを防ぐ第一歩です。
