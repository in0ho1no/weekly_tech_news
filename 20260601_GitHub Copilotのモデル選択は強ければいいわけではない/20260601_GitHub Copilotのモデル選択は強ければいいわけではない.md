# GitHub Copilotのモデル選択は強ければいいわけではない

## きっかけ

GitHub Copilotの課金体系について、2026年6月1日からリクエストベースから利用量ベースへ移行するという案内が出ています。  
利用量ベースでは、どのモデルを使うか、どれだけトークンを使うかによって扱いが変わります。  

ただし、この記事では細かい金額の話には踏み込みません。  
金額はプランや時期によって変わり得ますし、誤解があるとトラブルになりやすいからです。  

代わりに、少しだけ実用的な話をします。  

**Copilotのモデルは、常に一番強そうなものを選べばよいわけではありません。**

GitHub Docs「Models and pricing for GitHub Copilot」
<https://docs.github.com/en/copilot/reference/copilot-billing/models-and-pricing>
:::

## 強いモデルにも向き不向きがある

GitHubの公式ドキュメントでは、Copilotで使えるモデルをタスク別に整理しています。  
たとえば、軽い質問や単純な編集には高速なモデル、複雑なデバッグや設計判断にはじっくり考えることが得意なモデル、というような分け方です。  

VS Codeのドキュメントでも、回答前に内部で時間をかけて考えるタイプのモデルは、複雑なリファクタリング、設計判断、複数段階のデバッグに向いていると説明されています。  
一方で、単純な質問や定型コードの生成では、深く考えさせても待ち時間が増えるだけで、大きな利点にならない場合があります。  

つまり、「強いモデル」は万能というより、重い仕事を任せるための道具です。  
ちょっとした確認に毎回一番重い設定を使うのは、電卓で済む計算に解析ソフトを立ち上げるようなものかもしれません。

## Effortも上げればよいわけではない

Copilot Chatでは、モデルによってThinking effortを選べる場合があります。  
これは、モデルに「回答前にどれくらい考えさせるか」を調整する設定です。  

高いEffortは、複雑な問題では回答の質を上げる助けになる可能性があります。  
一方で、待ち時間やトークン使用量は増えやすくなります。  
低いEffortは、軽い確認や短い編集では十分なことが多く、応答も軽くなります。  

公式ドキュメントでは、利用できるEffortや既定値はモデルやプロバイダーによって異なるとされています。  
また、VS Code側では多くの用途では既定値でうまく動くように設定している、と説明されています。  

そのため、現時点で「Claude Sonnet 4.6なら必ずMedium」「Claude Opus 4.6なら必ずXHigh」のような対応表を公式情報として断言することはできません。  
ここは、タスクの重さに応じて切り替えるもの、と捉えるのが安全です。  

## 使い分けの目安

たとえば、次のような使い分けから始めるとよさそうです。  

- エラー文の意味を聞く
- 小さな関数の書き方を相談する
- 既存コードの一部を説明してもらう
- READMEやコメントの文章を整える

このあたりは、まずAuto、または軽め・標準的なモデルとMedium程度のEffortから始めてよいと思います。  
Claude Sonnet 4.6のような汎用的なモデルを、必要以上に高いEffortへ上げずに使うイメージです。  

一方で、次のような場面では強いモデルや高めのEffortを検討する価値があります。  

- 複数ファイルにまたがる原因不明のバグを追う
- 既存設計を崩さずに大きめのリファクタリングを考える
- ライブラリ選定やアーキテクチャのトレードオフを比較する
- ログ、設定、コードをまとめて読ませて原因を推定する

このような場合は、Claude Opus系やGPT-5系など、じっくり考える作業に向いたモデルを選ぶ意味が出てきます。  
それでも、最初から常にXHighにするのではなく、MediumやHighで様子を見て、足りなければ上げるくらいでよいでしょう。  

:::tip
迷ったら、まずAutoを使うのも手です。
GitHubの説明では、Auto（VS CodeのCopilot Chat）はモデルの稼働状況・可用性に加えてタスクの複雑さも見て、適したモデルへ振り分ける仕組みです。
このとき、プランに含まれないモデルや、管理者ポリシーで除外されたモデル、倍率が1を超える重いモデルは選ばれないため、知らないうちに高コストなモデルが使われる心配は基本的にありません。
ただし、選ばれるモデルはプランや組織のポリシーに左右されます。また、稼働状況とタスクの複雑さを組み合わせて選ぶ挙動はVS Codeでの提供で、他のIDEでは可用性・信頼性ベースの選択になります。
:::

## モデル選択は「質問の仕分け」

Copilotを使っていると、つい「一番賢いモデルに投げれば安心」と考えたくなります。  
でも実際には、質問には軽いものと重いものがあります。  

軽い質問には軽い道具を使う。  
重い判断には強い道具を使う。  
足りなければ、そこで一段上げる。  

それくらいの使い分けで十分です。  

大事なのは、モデル名の強さに引っ張られすぎないことです。  
Copilotをうまく使うというのは、強いモデルを選び続けることではなく、今の作業にちょうどよい相手を選ぶことなのだと思います。
